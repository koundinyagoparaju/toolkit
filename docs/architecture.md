# Architecture

How the pieces fit, and the byte-level contracts between them. For the
conceptual model see [concepts.md](concepts.md).

## Repository layout

```
crates/core          the contract: DataValue + coercions, Manifest/ports/
                     options, Tool trait + Registry, chain schema + the
                     push execution engine, StreamSession, the wasm ABI
crates/packs/*       tool implementations (text, image, crypto, data)
crates/cli           `toolkit` binary: links packs natively, fills entropy
                     from the OS RNG, streams chunked IO
web/                 Svelte app: catalog/tool pages/builder generated from
                     manifests; JS twins of the engine and ABI loader
chains/              the chain library (pure data)
scripts/             build-web-assets.sh, install.sh
.github/workflows/   ci (test+lint+audit), build (5-platform matrix),
                     release (tag -> binaries + SHA256SUMS), pages (deploy)
```

One tool implementation serves both frontends: the CLI links the Rust
directly; the browser fetches the same crate compiled to
`wasm32-unknown-unknown`. There is no server anywhere.

## The pack ABI

Each pack's wasm module exports the same functions, generated by
`toolkit_core::export_pack_abi!(crate::registry)`. The protocol is
deliberately tiny and language-agnostic; the JS loader
(`web/src/lib/wasm.js`) implements the other side in ~150 lines.

**Memory**: the caller allocates request buffers with `tk_alloc`, copies
bytes in, and frees with `tk_dealloc`. Functions return buffers packed as
`u64 = (ptr << 32) | len`; the caller copies out and frees. The module
imports **nothing** from the host — this is the sandbox guarantee.

**Framing** (shared by requests and responses):

```
[u32 little-endian header length][header JSON][payload bytes]
```

**One-shot execution**:

```
tk_manifests() -> u64                      # JSON array of tool manifests
tk_run(ptr, len) -> u64
  request header:  {"tool": "...", "options": {...},
                    "inputs": [{"port": "input", "type": "text", "len": 5}, ...]}
                   # one entry per input value, payloads concatenated in
                   # header order; repeated port names = multi-port values
  response header: {"ok": true, "output": {"type": "...", "format": "..."}}
                   | {"ok": false, "error": "..."}
```

**Streaming** (stateful sessions in a per-pack slab):

```
tk_stream_open(ptr, len) -> u64            # header {"tool", "options"};
                                           # response {"ok", "handle": N}
tk_stream_update(handle, ptr, len) -> u64  # header {"port", "index"} + chunk;
                                           # response payload = emitted bytes
tk_stream_end_input(handle, ptr, len)      # header {"port", "index"}
tk_stream_finish(handle) -> u64            # final bytes; frees the session
tk_stream_abort(handle)                    # frees without finishing
```

Errors poison the handle (the session is dropped; further calls report an
invalid handle). A pack written in Go/Zig/C that exports these symbols and
speaks this framing would drop into the web app unchanged.

## Execution engine

`Chain::execute_streaming` (mirrored by `web/src/lib/dagRun.js`) is a
push-based dataflow and the **only** engine — buffered execution is the
same engine fed one chunk with outputs retained:

1. **Wiring**: each node gets *slots*, one per expected input value,
   grouped per port in edge-declaration order (that order defines
   multi-port value order). Entropy slots are primed from the driver's
   entropy source before any input flows.
2. **Events**: `Chunk(node, slot, bytes)` and `End(node, slot)` flow
   through a FIFO queue. Streaming nodes pass chunks through their session
   and emit downstream immediately. Non-streaming nodes ("reservoirs")
   accumulate into their slots; when every slot has ended, the tool runs
   once via the same `run_tool` path used everywhere, and its output is
   pushed onward — with its *runtime* metadata (e.g. actual image format)
   written into downstream slot metas.
3. **Sinks** (nodes with no outgoing edges) deliver incrementally through
   a callback: the CLI writes chunks to stdout/files as they arrive.

Because reservoirs buffer only at their own inputs, memory is bounded by
the largest single buffered step, never the sum of intermediates.

## Drivers

The core is deliberately inert: anything environmental is injected by a
*driver* —

| Concern | CLI | Web |
|---|---|---|
| chain input | stdin/file, 1 MB chunks | typed value, or `file.stream()` for large files |
| entropy ports | `getrandom` (OS RNG) | `crypto.getRandomValues` |
| sink output | stdout / files, incremental | buffered for display (+ capped previews for streamed intermediates) |
| tool discovery | native registry | `manifests.json` (generated by `toolkit manifests`, includes the coercion matrix so the JS validator can't drift from core) |

## Build & release pipeline

- `scripts/build-web-assets.sh`: packs → wasm, catalog → `manifests.json`,
  chain library → `web/public/chains/`.
- CI (every PR): fmt, clippy `-D warnings`, full tests, wasm+web build,
  `cargo audit` (this gate has caught real RUSTSEC advisories).
- Tag `v*`: five-platform static binaries + `SHA256SUMS` published as a
  GitHub Release — the exact artifacts `scripts/install.sh` verifies.
- Push to `main`: the web app deploys to GitHub Pages. The CSP lives in a
  `<meta>` tag because static hosts can't send headers; hash routing means
  no rewrites; all asset paths are relative so the mount point is
  irrelevant.

## Reproducing a release

Releases are built with a pinned toolchain (`rust-toolchain.toml`), locked
dependencies, normalized path prefixes, and deterministic tar/gzip — so
you can rebuild a release binary yourself and compare hashes instead of
trusting ours. For the Linux x86_64 artifact:

```sh
docker run --rm rust:1.96.0 bash -c '
  apt-get update -qq && apt-get install -y -qq musl-tools >/dev/null &&
  git clone --depth 1 --branch <TAG> https://github.com/koundinyagoparaju/toolkit /src &&
  cd /src && rustup target add x86_64-unknown-linux-musl &&
  export RUSTFLAGS="--remap-path-prefix=$PWD=/build --remap-path-prefix=$HOME=/remapped-home" &&
  cargo build --release --locked --target x86_64-unknown-linux-musl -p toolkit-cli &&
  sha256sum target/x86_64-unknown-linux-musl/release/toolkit'
```

Compare against the `toolkit` binary extracted from the release tarball.
CI enforces this property on every PR (two clean builds in different
directories must hash identically), and releases additionally carry GitHub
build-provenance attestations:
`gh attestation verify toolkit-linux-x86_64.tar.gz --repo koundinyagoparaju/toolkit`.

Applies to releases after v0.1.0 (earlier builds predate the normalized
flags).
