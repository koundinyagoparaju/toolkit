# Concepts

The mental model behind toolkit, from the inside out. Read this once and
the rest of the codebase should feel predictable.

## Values and types

Everything flowing between tools is a **`DataValue`** with one of four
types:

| Type | Holds | Example producers |
|---|---|---|
| `text` | UTF-8 string | base64-encode, json-format |
| `bytes` | raw bytes | base64-decode, random-bytes |
| `json` | structured value | jwt-decode, url-parse |
| `image` | *encoded* image bytes + format tag | image-resize, qr-generate |

Two deliberate simplicities:

- **Images stay encoded.** An `image` value is png/jpeg/... bytes, not
  pixels. Image tools decode on entry and re-encode on exit. This keeps the
  ABI and every non-image tool ignorant of pixel formats.
- **There are no list or nested types.** Variable arity lives on input
  *ports* (below), never in the type system.

### Coercions

Types connect more often than they match exactly, so a fixed **coercion
matrix** (`crates/core/src/data.rs`) defines the sanctioned conversions:

| from \ to | text | bytes | json | image |
|---|---|---|---|---|
| **text**  | — | UTF-8 bytes | parse | ✗ |
| **bytes** | must be UTF-8 | — | parse | tag as image (format sniffed) |
| **json**  | serialize | serialize | — | ✗ |
| **image** | ✗ | encoded bytes | ✗ | — |

Chain edges are validated against this matrix *statically*; the conversion
itself happens at runtime and can still fail on the actual value (bytes
that aren't UTF-8, text that isn't JSON). Everything outside the matrix is
rejected before anything runs.

## Tools

A tool is a pure function described by a **manifest**:

```text
name        stable kebab-case slug ("base64-decode") — used in chains & CLI
label       human name shown in UIs
description one or two sentences; say what AND why-it-matters
keywords    search terms for the catalog
inputs      the input ports (below)
output      one DataType
streaming   true if the tool implements a StreamSession
options     typed option specs (below)
```

Everything downstream is generated from the manifest: the web catalog
entry, the tool page form, the CLI flags, chain type-checking. Adding a
tool never requires touching the CLI or the web app.

**Purity rules**: no filesystem, no network, no clock, no ambient
randomness. A tool's output is a function of its inputs and options,
nothing else. (Randomness, where needed, arrives as an input — see
entropy ports.)

### Input ports

Tools declare *named, typed* input ports. Four shapes cover everything:

| Shape | Declaration | Example |
|---|---|---|
| sole | `InputSpec::sole(DataType::Text)` | almost every tool; port named `input` |
| named | `InputSpec::named("first", Image)` ×N | image-merge (`first`, `second`); text-diff (`old`, `new`) |
| multi | `InputSpec::named("documents", Text).multi()` | doc-merge: accepts 1..N ordered values |
| entropy | `InputSpec::entropy()` | uuid, password-gen: driver-filled randomness |

- Ports have distinct **roles**; that's why merge tools use named ports
  instead of an array type — "first" vs "second" is visible and
  type-checked, and the chain builder renders one labeled dot per port.
- A **multi** port receives an ordered list; order is defined by edge
  declaration order in a chain, or the order of repeated `-i` flags in the
  CLI.
- An **entropy** port is invisible in UIs and auto-filled by the *driver*
  (CLI: OS RNG; web: `crypto.getRandomValues`; chain engines prime it)
  with `ENTROPY_LEN` (1024) random bytes. The tool stays deterministic
  given its inputs — wire fixed bytes into the port in a chain and you get
  reproducible output. This also preserves the wasm sandbox's zero-import
  property: randomness enters as auditable data, not through a host call.

### Options

Options are the tool's knobs: `string`, `integer{min,max}`,
`float{min,max}`, `bool`, `enum{values}`, each optionally `required` or
carrying a `default`. Core validates and normalizes them (defaults filled,
unknown keys rejected, ranges enforced) before `run` is called — tools
read validated values and never re-validate. The same specs render as the
web form and as CLI `--set key=value` flags, and chains reuse the format
for their declared params.

## Packs

Tools ship in **packs** — one Rust crate each (`crates/packs/{text,image,
crypto,data}`), compiled two ways:

- **natively** into the CLI (plain Rust linking, no wasm at runtime);
- **to `wasm32-unknown-unknown`** for the browser, one `.wasm` module per
  pack, lazy-loaded on first use and cached offline by the service worker.

Grouping matters for the web: codecs compile once per pack, and a visitor
who never touches image tools never downloads the image pack. The wasm
side of a pack is generated by one macro line
(`toolkit_core::export_pack_abi!(crate::registry)`); the byte protocol is
documented in [architecture.md](architecture.md) and is deliberately
language-agnostic — a future pack could be written in anything that
compiles to wasm and speaks the same framing.

## Chains

A **chain** is a DAG of tool invocations, stored as JSON
(`chains/*.json`), shared by the CLI, the web builder, and shareable URLs
(the URL encodes the definition, never data). Rules, all enforced by
validation before execution:

- node ids unique; every edge references existing nodes; no cycles;
- **fan-out** is allowed (one output feeding many nodes);
- a *single* port accepts at most one incoming edge; a **multi** port
  accepts many (values ordered by edge declaration order);
- a node is either **fully wired** (every non-entropy port has an edge) or
  an **entry node** (no incoming edges) — entry nodes receive the chain's
  input on every non-entropy port;
- edges are type-checked against the coercion matrix, per target port.

**Params** make a chain a first-class, callable unit: named, typed knobs
(same spec format as tool options) with `maps` writing their value into
node options. Precedence: invocation `--set` > chain file > tool defaults.

## Streaming

Some tools are **byte transducers** (hash, base64, hex, url, gzip,
doc-merge): they can process input incrementally. Those implement a
`StreamSession` — chunks tagged `(port, index)`, an `end_input` per value,
a final `finish` — and set `streaming: true`. Two properties keep this
honest:

1. **The buffered path is derived from the session** (`buffered_run`), so
   a streaming tool has exactly one implementation and the modes cannot
   diverge.
2. **There is one chain engine, not two.** Chains always execute as a
   push-based dataflow: streaming nodes transform chunk-by-chunk;
   whole-value tools (images, JSON) are "**reservoirs**" that buffer at
   their own inputs and run once complete. Buffered execution is the same
   engine fed a single chunk. Memory is bounded by the largest reservoir —
   an all-streaming chain runs in constant memory at any input size.

Whole-value tools don't implement sessions and lose nothing: the engine
buffers exactly at their boundary and streams everywhere else.

## The trust model (why all of the above)

Every design above serves one promise: *data never leaves the device, and
you don't have to take our word for it.* Static site + strict CSP
(`connect-src 'self'`) means even compromised code can't exfiltrate; the
wasm sandbox imports nothing from the host; the CLI contains no network
code; dependencies are pure-Rust, pinned, and audited in CI; and
everything a visitor runs is built from this repository through public,
reviewable pipelines. When you contribute, you're extending that promise —
which is why tools enter through reviewed PRs and why the purity rules are
non-negotiable.
